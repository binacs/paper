# VUDA: Breaking CUDA-Vulkan Isolation for Spatial Sharing of Compute and Graphics on the Same GPU

## TL;DR

VUDA breaks CUDA-Vulkan execution isolation to enable spatial sharing of compute and graphics on the same GPU, achieving up to 85% higher throughput than temporal sharing for embodied-AI workloads.

## Problem & Motivation

Embodied AI simulation environments interleave physics simulation (CUDA) and photorealistic rendering (Vulkan) on a single GPU. Two key scenarios—simulation data generation and RL training—can naturally execute simulation and rendering phases concurrently, offering an opportunity to improve GPU utilization via spatial multiplexing. However, CUDA and Vulkan create separate GPU contexts with channels bound to different scheduling groups, confining compute and graphics to mutually exclusive time slices (execution isolation). Existing spatial-sharing techniques are limited to the CUDA ecosystem, while temporal sharing underutilizes resources.

## Key Ideas

- Observation: CUDA and Vulkan execution paths converge to a common channel primitive at the driver/hardware level.
- Observation: Their virtual-address spaces are inherently disjoint, making safe page-table merging feasible without remapping.
- Thin API for developers to annotate co-schedulable CUDA streams.
- Channel redirection: redirect CUDA channels into Vulkan's scheduling domain to enable spatial sharing.
- Page-table grafting: unify address spaces to eliminate data copying on the critical path.

## System Design

- VUDA exposes a thin API for developers to annotate CUDA streams that can be co-scheduled with Vulkan graphics workloads.
- Channel redirection: VUDA redirects CUDA channels into Vulkan's scheduling domain, allowing them to be scheduled concurrently.
- Page-table grafting: VUDA merges the page tables of CUDA and Vulkan contexts, enabling safe access to each other's memory without remapping or copying.
- The design eliminates all data copying on the critical path, enabling zero-copy sharing.

## Evaluation

Experiments on representative embodied-AI workloads show VUDA delivers up to 85% higher throughput than temporal-sharing baselines, while improving GPU utilization and reducing end-to-end latency.

## Takeaways

- CUDA and Vulkan share a common channel primitive at the driver/hardware level, enabling spatial sharing despite different APIs.
- Page-table grafting is safe because CUDA and Vulkan address spaces are disjoint.
- Spatial sharing can significantly improve GPU utilization for interleaved compute-graphics workloads.
- VUDA's approach requires only minor API annotations from developers.

## Related Work

Existing spatial-sharing techniques are limited to the CUDA ecosystem (e.g., CUDA streams). Temporal-sharing approaches (e.g., time-slicing) underutilize resources. VUDA is the first to break CUDA-Vulkan isolation for spatial sharing.

## Open Questions

How does VUDA handle workloads with dynamic or unpredictable co-scheduling patterns? What is the overhead of page-table grafting for large address spaces? Can the approach be extended to other GPU APIs (e.g., OpenCL, DirectX)?
