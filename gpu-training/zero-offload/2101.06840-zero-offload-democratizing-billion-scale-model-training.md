# ZeRO-Offload: Democratizing Billion-Scale Model Training

## TL;DR

ZeRO-Offload enables training of models with over 13 billion parameters on a single GPU by offloading data and compute to CPU, achieving 40 TFlops/GPU on a V100 for a 10B parameter model without model changes.

## Problem & Motivation

Training large-scale deep learning models (billions of parameters) typically requires multiple expensive GPUs and complex model refactoring, limiting access to only a few organizations. Existing frameworks like PyTorch run out of memory on a single GPU for models larger than ~1.4B parameters, and scaling to multiple GPUs introduces high communication overhead and requires expertise.

## Key Ideas

- Offload optimizer states and gradients to CPU memory, keeping only forward/backward activations on GPU.
- Minimize data movement between CPU and GPU by batching transfers and overlapping compute with communication.
- Use CPU for optimizer update computation to reduce GPU memory pressure, while GPU handles the compute-intensive forward/backward passes.
- Design a partitioning strategy that balances memory savings and compute efficiency, ensuring CPU compute time does not become a bottleneck.

## System Design

ZeRO-Offload extends the ZeRO (Zero Redundancy Optimizer) approach. It partitions model states: optimizer states and gradients are stored on CPU, while parameters and activations remain on GPU. During training, the GPU performs forward/backward passes; gradients are transferred to CPU asynchronously. The CPU updates optimizer states and sends updated parameters back to GPU. Data transfers are overlapped with GPU computation to hide latency. For multi-GPU, ZeRO-Offload partitions data across GPUs and uses all-reduce for gradient synchronization, achieving near-linear scaling.

## Evaluation

Evaluated on GPT-2 and other large models up to 13B parameters on a single NVIDIA V100 GPU. Achieves 40 TFlops/GPU for a 10B parameter model, compared to 30 TFlops for PyTorch on a 1.4B model (largest possible without offloading). Scales to 128 GPUs with near-linear speedup. Combined with model parallelism, trains 70B parameter models on a single DGX-2 box (4.5x increase over model parallelism alone).

## Takeaways

- Offloading to CPU can dramatically increase trainable model size on a single GPU without sacrificing efficiency.
- Careful overlapping of CPU-GPU transfers and compute is key to maintaining throughput.
- ZeRO-Offload democratizes large model training, making it accessible to researchers with limited GPU resources.
- The approach is complementary to model parallelism and can scale to multiple GPUs.

## Related Work

ZeRO-Offload builds on the ZeRO optimizer (Rajbhandari et al., 2020) which reduces memory redundancy across GPUs. It differs by offloading to CPU memory rather than distributing across GPUs, enabling single-GPU training. Compared to model parallelism (e.g., GPipe, Megatron-LM), ZeRO-Offload requires no model changes and works with existing frameworks like PyTorch.

## Open Questions

- How does performance degrade when CPU memory bandwidth is limited (e.g., on consumer hardware)?
- Can the approach be extended to offload to other memory tiers (e.g., NVMe SSDs) for even larger models?
- What is the impact on training convergence due to potential staleness from asynchronous updates?
- How does it compare to newer offloading techniques like ZeRO-Infinity or DeepSpeed's CPU offload?
